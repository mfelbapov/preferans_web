# Plan: Phoenix Project Setup for Preferans Web UI (Step 1)

## Context

The Preferans C++ game engine (rules, AI, training) exists at `/Users/mfelbapov/Projects/preferans/`. The next phase is building a web UI using Elixir Phoenix + LiveView so humans can play against AI. The C++ engine will run as an Erlang Port (JSON over stdin/stdout) — but that protocol doesn't exist yet in the C++ side.

**This plan covers Step 1 only**: project scaffolding, auth, database schema, stub routes. No game logic, no C++ integration, no card UI yet.

## Decision: Project Location

Separate repo at `/Users/mfelbapov/Projects/preferans_web/`. Clean separation of C++/Elixir tooling, independent git history.

## Environment

- Elixir 1.18.4, Erlang/OTP 28, Phoenix installer 1.8.1
- PostgreSQL (local)
- Pattern reference: ohmyword project at `/Users/mfelbapov/Projects/ohmyword/`

## Execution Steps

### 1. Generate Phoenix project
```bash
cd /Users/mfelbapov/Projects
mix phx.new preferans_web --live
```
Generates: Phoenix 1.8, LiveView, Tailwind, esbuild, Bandit, PostgreSQL.

### 2. Generate auth
```bash
cd preferans_web
mix phx.gen.auth Accounts User users --live
```
Generates: User schema, tokens, session auth, registration/login LiveViews, `on_mount` hooks. Pattern matches ohmyword's auth setup.

### 4. Create Matches table
A "match" = full game session (many hands until someone burns all bule).

| Field | Type | Purpose |
|-------|------|---------|
| `status` | `string` | `in_progress`, `completed`, `abandoned` |
| `mode` | `string` | `3h`, `2h1ai`, `1h2ai` |
| `max_refes` | `integer, default: 2` | Refe limit |
| `initial_bule` | `integer, default: 10` | Starting bule |
| `final_bule` | `{:array, :integer}` | End state [p0, p1, p2] |
| `final_refe_count` | `{:array, :integer}` | End state |
| `winner_seat` | `integer` | 0, 1, or 2 |
| `total_hands` | `integer` | Hand count |
| `player_0_id` / `1` / `2` | `references(:users)` | Nullable for AI seats |
| `player_0_type` / `1` / `2` | `string` | `"human"` or `"ai"` |

Three separate player columns (not a join table) because Preferans is always exactly 3 seats.

## Design Decisions & Tradeoffs

1. **Bitsets as integers in Postgres** — compact, not queryable by individual card. Fine because we never need SQL-level card queries; analysis loads into Elixir.

2. **`action_history` as `{:array, :map}`** — stores full replay data as JSON. Could be a separate `actions` table for queryability, but that adds complexity for a feature (replay) that loads the whole hand anyway.

3. **Three player columns vs join table** — denormalized but simpler. Preferans is always exactly 3 seats; a join table adds overhead for zero flexibility benefit.

4. **No C++ port stub** — the C++ engine has no JSON protocol yet. Stubbing it would create a fake interface that needs rewriting. Better to add it in Step 2-3 when we define the real protocol.

5. **Glicko-2 fields on User directly** — simpler than a ratings table. Rating history can be added later if needed.
